Add Wiki to Group

Alex Nilson

Actors: Group owner, server

Goals: Add a group wiki for collaboration, communication, etc. within the group

Preconditions: Group owner owns a group, group does not yet have a wiki, group owner is logged into gus.

Summary: A wiki can be added to a group for collaboration, documentation, communication, entertainment, or anything else the group wants. Users that are members of the group will be able to edit the wiki freely and other members will be able to view the changes.

Related use cases: Edit wiki, delete wiki

Steps:

  1. Group owner opens the group management menu.
  2. Group owner checks the box to enable a group wiki and adjusts permissions for the wiki.
  3. Group owner saves changes.
  4. Server adds a pre-generated "main" page to the wiki and adds a link to the group page.

Alternative:

  • During group creation, group owner can start at step 2 and continue as normal.
  • At step 3, group owner can choose not to save changes and the process stops.

Fail: If connection times out the user will be informed by their browser and will have to try to add the wiki again. If an error occurs on the server side, the user will receive a message indicating what went wrong and whether they should try again or alert the system administrator.

Edit Wiki

Alex Nilson

Actors: User, server

Goals: Edit the content of the group wiki

Preconditions: Group has a wiki, user is a member of that group, user has permission to edit the wiki, user is logged in.

Summary: If the user wants to add content to the wiki, they can edit wiki pages or add new pages to the wiki.

Related use cases: Add wiki to group, delete wiki

Steps:

  1. User finds the page of the wiki they want to edit.
  2. User clicks the "edit" button on the page.
  3. User uses the built in editor to adjust the contents of the page.
  4. User saves changes.
  5. Server updates contents of the page to match the user's edits and takes the user back to the edited page.

Alternative:

  • At step 1, user can choose to create a new page in the wiki instead of editing an existing one.
  • At step 4, user can choose to preview changes before they're made to the wiki.
  • At step 4, user can choose to cancel making changes.

Fail: If connection times out the user will be informed by their browser and will have to try to edit the wiki again. If an error occurs on the server side, the user will receive a message indicating what went wrong and whether they should try again or alert the system administrator. If the user doesn't have permission to edit the wiki, the server will let them know and prevent them from editing the wiki.

Update Document

Colby Blair

Actors: Editing user, group web page, other group users

Goals: User makes change document, possibly with many users at one time

Preconditions: User must me logged in, user must meet group criteria for updating document.

Summary: User wants to edit a GUS document, and clicks the document link on the group's document section of their website to view and edit the document

Related use cases:

Steps:

  1. User navigates to document section of group web page
  2. User clicks on document link
  3. Group web page displays document content
  4. User types to make document changes
    • If user is in an online mode with low enough latency, changes are made live and saved as user types.
    • If not, file is locked in the database in case it is shared. Other users can communicate in document chat, and can view document.

  6. Document is saved and any locks are freed.

Postconditions: Document may have changed name or content.

Alternative: User deletes document, user locks file but doesn't save changes.

Fail: User opens document, but loses internet connection. Upon reconnection, user can try to save document, or merge if changes have been made in the mean time. If conflicts arise, a diff can be displayed and user can choose which changes to keep.

Sync with server

Leah Wegner

Actors: (Admin, Group Leader, or Group Moderator) and server

Goals: Syncs the user's program which they used to edit somthing in one of their groups on their own computer with the server's version of the group. Also invoked immediately after something has been changed in the online interface.

Preconditions: The Actor possesses a version of gus installed on their current computer. They have a connection to the server. The server is available.

Summary: Syncs the user's program which they used to edit somthing in one of their groups on their own computer with the server's version of the group. Also invoked immediately after something has been changed in the online interface.

Related use cases: all the user-interface use-cases that will eventually connect to the server. For example: Add Blog, Add User, and Upload Document.

Postconditions: The server has been updated to reflect the changes.

Steps

  1. User pushes a button on the interface to connect to the server.
  2. gus program connects to the server.
  3. gus program uploads changed content.
  4. Server confirms the upload, and returns a message to the user showing what was changed.

Alternatives:

  1. Throughout step 2 and 3, the user may cancle the connection, resulting in the server reverting to the state it was in prior to any changes that had started to be made.

Fail:At step 2, the connection could fail, and no changes will be made to the server. An error message will inform the user of the problems.

Add user note

Leah Wegner

Actors: (Group Leader, or Group Moderator) and server

Goals: Create a group-specific note about a user.

Preconditions: The Actor has at least "Moderator" level permissions within the group they are affecting, and the user to attach a note to is a member of the group.

Summary: This function is used when a group leader or moderator would like to keep track of some information about a specific user. For example, whether or not they have paid their dues, or when they can meet. This note is only visible to group leaders and group moderators.

Related use cases: Edit user note, Remove user note, Sync with server

Postconditions: A note is created and attached to the account within the scope of the group.

Steps

  1. Actor pushes a button on the interface to add a note to a user.
  2. Actor selects a user from a list, or types in the user's name.
  3. gus program creates a note and links it to the user.
  4. gus program calls Sync with server.
  5. gus program calls Edit user note.

Alternatives:

  1. The Actor may cancel during step 2, 4 or 5.

Fail:At step 4 or 5, the connection with the server could fail, and no changes will be made. An error message will inform the user of the problems.

Set RSVP limit for event

Leah Wegner

Actors: (Group Leader, or Group Moderator) and server

Goals: Set the number of people who are allowed to RSVP to an event before the waitlist system must be envoked.

Preconditions: The event must exist, and be in the future.

Summary: This function is used to set how many people are allowed to attend an event that has limited space such as a formal lunch. If this function is not called, there will be no limit.

  5. Actor hits ok.
  6. gus program edits the event.
  7. gus program calls Sync with server.

Alternatives:

  1. The Actor may cancel at step 1 or 5.
  2. If called by an internal function, skip step 1.

Fail:At step 7 the connection with the server could fail, and no changes will be made. An error message will inform the user of the problems.
